# 常见问题

  1. GET 方法的长度限制？

对于上文表格中提到的 “数据长度的限制”，其实 HTTP 协议是没有长度限制的，限制的是浏览器或服务器，不同浏览器对 URL 长度的限制也不一样，建议是不超过 2048 个字符。

IE：对 URL 的最大限制为 2083 个字符，若超出这个数字，提交按钮没有任何反应。
Firefox：对 Firefox 浏览器 URL 的长度限制为：65536 个字符。
Safari：URL 最大长度限制为 80000 个字符。
Opera：URL 最大长度限制为 190000 个字符。
Google (chrome)：URL 最大长度限制为 8182 个字符。
Microsoft Internet Information Server (IIS)：n 能接受最大 url 的长度为 16384 个字符。

  1. GET、POST 的数据一定是分别放在 url 和 body 中？

其实并不然，HTTP 规范中并没有强制要求；这两种方法本质上是 TCP 连接，既可以使用 POST 方法在 URL 上写参数，也可以使用 GET 方法在 Body 写参数；

  • 对于 POST 方法，在 URL 上写参数好理解，实际上就是携带空请求体，对一个具体的网址发送请求；
  • 对于 Get 方法则进行了一次试验：


可见，结果证明 GET 方法携带请求体 是可行的。

但是在开发过程中，仍旧推荐按平常的开发规范来实施，因为前后端都需要互相配合，server 也能拒绝带 body 的 get 请求，现在也有部分如 okHTTP3 、 fetch 等不支持 GET/HEAD 请求携带请求体。

RFC7231：“A payload within a GET request message has no defined semantics; sending a payload body on a GET request might cause some existing implementations to reject the request.”

  1. POST 方法产生两个 TCP 数据包？

有些文章中提到，post 会将 header 和 body 分开发送，先发送 header，服务端返回 100 状态码再发送 body。
HTTP 协议中没有明确说明 POST 会产生两个 TCP 数据包，经他人测试 Firefox 上的 POST 请求只会抓到一个 TCP 包；
在 MDN 中提到，只有发送请求中的首部携带了 Expect: 100-continue ，才会检查请求首部，而且也有人测试 Chrome 发现，header 和 body 不会分开发送；
所以，header 和 body 分开发送是部分浏览器或框架的请求方法，不属于 post 必然行为。
